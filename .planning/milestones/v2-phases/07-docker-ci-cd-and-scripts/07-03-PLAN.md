## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub runner → Oracle VPS | SSH deploy key crosses from CI secrets to a remote host shell |
| git-archived source → on-box image build | Source shipped to the box becomes the running production image |
| oven/bun base image → deployed container | Upstream base layers run in production |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Information Disclosure | DEPLOY_SSH_KEY handling in deploy.yml | accept | Existing correct control (not modified here): key written to `~/.ssh/deploy_key` with `chmod 600`, sourced from `secrets.DEPLOY_SSH_KEY`, never echoed; StrictHostKeyChecking=accept-new |
| T-07-07 | Elevation of Privilege | auto-deploy-on-push trigger | mitigate | `push:` trigger kept commented (workflow_dispatch only) so deploys are deliberate, human-initiated; not silently enabled |
| T-07-08 | Information Disclosure | secrets in image / compose | accept | `docker-compose.box.yml` uses `env_file: .env` (runtime injection); Plan 01 Dockerfile bakes no secrets — no change needed |
| T-07-SC | Tampering | base-image / dep supply chain on box build | mitigate | Pinned `oven/bun:1.3.14-slim` + `--frozen-lockfile` (Plan 01) ensure the on-box build is reproducible; no `latest` tag |
