<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer/CI shell to `package.json` scripts | Local and future CI callers execute script strings that resolve `bun`, `npm`, `node`, and root `concurrently` from PATH and installed dependencies. |
| Public package registry to lockfiles/node_modules | Bun and npm resolve dependency tarballs from the registry into generated lockfiles and local installs. |
| Generated lockfiles to committed source | `bun.lock` and `package-lock.json` become trusted source artifacts used by future developers and CI. |
| Bun default path to npm/Node fallback path | Operational fallback depends on exact `:node` scripts and npm lockfiles remaining in sync with Bun defaults. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | `package.json` scripts | mitigate | Use exact script assertions in Task 1 so Bun defaults and npm/Node fallback aliases cannot silently drift. |
| T-06-02 | Tampering | `bun.lock`, `package-lock.json`, `client/bun.lock`, `server/bun.lock` | mitigate | Generate and commit lockfiles, then verify with `bun install --frozen-lockfile` and `npm ci --ignore-scripts` for every package root. |
| T-06-03 | Repudiation | Future dependency changes | mitigate | Add `lockfiles:refresh` per D-02 so the repo has an executable record of the required dual-lockfile maintenance action. |
| T-06-04 | Denial of Service | Local install/fallback commands | mitigate | Preserve exact npm fallback lockfiles and validate both Bun and npm install paths before completion. |
| T-06-SC | Tampering | Package-manager/lockfile supply chain | mitigate | Promote only the already-present `concurrently` package at the same `^9.2.0` spec, freeze it in both root lockfiles, retain client/server package-locks, and verify all generated lockfiles before completion. |
</threat_model>
