<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser upload -> Express JSON | Untrusted base64 image and claimed metadata enter the server |
| Image decoder -> native sharp/libvips | Complex untrusted image formats reach native parsing code |
| Authenticated user -> company document | Tenant identity and current role govern reads/writes |
| Company branding -> browser data URL | Stored bytes return to users in the same tenant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | extract/save routes | mitigate | Re-query current user through `requireCompanyAdmin`; integration test asserts member 403 |
| T-02-02 | Information Disclosure | branding GET | mitigate | Scope company lookup only by authenticated `req.user.companyId`; cross-tenant integration assertion |
| T-02-03 | Denial of Service | oversized/decompression-bomb image | mitigate | 8 MB transport cap, 5 MB decoded cap, sharp `limitInputPixels`, 1024x1024 resize |
| T-02-04 | Tampering / XSS | malicious SVG or spoofed content type | mitigate | Decode every accepted format with sharp and persist/serve normalized PNG only; never serve original SVG |
| T-02-05 | Information Disclosure | audit logs | mitigate | Audit only file metadata and changed fields; never log base64/binary content |
| T-02-SC | Tampering | new npm dependencies | mitigate | Exact versions already proven on native arm64 in Phase 1; dual lockfiles committed |
</threat_model>
